TransitAI UTM — Administrator Manual

Web-Based Campus Transport Chatbot (Prototype) · MECS0033-52 Artificial Intelligence · Group 5

Scope. This manual documents the prototype's components, configuration and operating procedures (project rubric: Admin Manual criterion). The prototype is fully client-side — HTML + CSS + vanilla JavaScript, with the knowledge base embedded in js/kb.js. No server, database, API key, or build step. Public route names/sequences are aligned to public UTM/KDOJ route listings where available; timetable/ETA values and missing landmark mappings are labelled prototype simulation.

1. Overview

TransitAI UTM answers UTM campus shuttle questions, reproducing the report's AI pipeline: intent recognition → knowledge retrieval (RAG-style) → response generation, plus a resolution-refutation engine for delay notifications.

4. Configuration notes

All configuration is plain data in js/kb.js. Edit, save, reload the browser — no rebuild.

4.1 Add / edit a bus stop (KB.stops)

{ id: "fkm", name: "FKM (Faculty of Mechanical Engineering)",
  aliases: ["fkm", "mechanical", "mekanikal"],
  status: "prototype mapping",
  landmark: "FKM main block.",
  facilities: ["Lecture halls", "Café"],
  walking: "Stop is at the FKM drop-off bay." }

4.2 Add / edit a route (KB.routes)

{ id: "route_g", name: "BAS G1/G2/G3 — KTR/KTHO/KTDI to SKT",
  stops: ["ktr", "ktho", "ktdi", "n24", "skt", "p19", "cp"],
  operating: { start: "07:00", end: "18:30" }, // simulated
  headway: 20,
  note: "Public route sequence; timings are simulated.",
  source: "Public UTM/DVC/KDOJ route listing; current timing simulated/unverified." }

Timetable times are derived from operating + headway by nextDeparture() for a repeatable POC.

4.3 FAQ / 4.4 Intents / 4.5 Subscriptions

  • FAQ: add {q, keywords, a} to KB.faqs.
  • Intents: edit the CUES table in js/intent.js (strong = 2 pts, weak = 1 pt). Keep ultra-common words out of strong.
  • Subscriptions: KB.subscriptions holds WantsRouteAlert(user, route); demo user is ali (Chat.DEMO_USER).

4.6 Demo time control

The app defaults to 10:00 so schedule cards do not depend on the real current time. In Staff Demo, choose 08:00, 10:00, 13:00, 17:30, 20:30, or Live device time.

5. Operating procedure — delay alert with logic proof

  1. Open the 🛠️ Staff Demo tab.
  2. Toggle a route (e.g. BAS A1/A2 — KP to Lingkaran Ilmu) to delayed.
  3. The proof panel runs Resolution.prove("ali","route_a") and prints P1–P4, the negated goal, and R1→R3→⊥, proving NotifyUser(ali, route_a, delay) (report Figure 5.2).
  4. Open 🔔 Alerts to see the user notification.
  5. Toggle back to clear the delay.

Cannot misfire: the derivation only runs when the facts hold (subscribed and delayed); inputs come from app state, never from unparsed free text.

8. Limitations (prototype scope)

  • Sample data only; arrival is simulated (no live GPS feed).
  • Retrieval is token-overlap, not a real vector DB.
  • Resolution engine scoped to the delay-notification clause set.
  • Notifications appear in-app; no real push delivery.
